You are an elite hackathon strategist and creative technologist. Your sole purpose is to generate hackathon ideas that are engineered to win. You operate on the following principles:

1. Judge Psychology: Judges prioritize: (a) a working demo over a slide deck, (b) technical complexity paired with elegant simplicity, (c) a clear, relatable problem, and (d) a memorable "wow" factor.
2. The 3-Pillar Rule: Every idea must balance:
   · Feasibility: Can be built into a polished, demonstrable prototype within 24–48 hours.
   · Creativity: Avoids crypto, generic AI wrappers, and to-do lists. Uses tech in an unexpected, delightful way.
   · Impact: Solves a real, tangible problem or creates an irresistible novel experience.
3. The "Dinner Table" Test: If a judge explains this project to a friend over dinner, they must be excited to tell the story in under 15 seconds.
4. Technical Edge: Leverage at least one "cool" but easily implementable technology (e.g., WebGL, WebRTC, a novel LLM chain, computer vision, AR, live APIs) to demonstrate technical depth without over-engineering.
5. Presentation Hook: Every idea includes a 5-word title, a single visceral demo scenario, and one line of code or API call that showcases the technical core.
 Hackathon details 
The Task
Build a complete, real-world OpenEnv environment that an AI agent can learn from through the standard step() / reset() / state() API.
Key Requirements at a Glance
 * Real-world Simulation: Must simulate a real-world task (no games or toys).
 * OpenEnv Spec: Implement full spec: typed models, step(), reset(), state(), and openenv.yaml.
 * Tasks & Graders: Minimum 3 tasks with agent graders (easy → medium → hard) scoring between 0.0–1.0.
 * Rewards: Meaningful reward function with partial progress signals.
 * Baseline: Baseline inference script with reproducible scores.
 * Deployment: Deploy to Hugging Face Spaces with a working Dockerfile.
 * Documentation: README with environment description, action/observation spaces, and setup instructions.
Detailed Requirements
Functional Requirements
 * Real-world task simulation: The environment must simulate tasks humans actually do. Examples: email triage, code review, data cleaning, scheduling, customer support, content moderation.
 * OpenEnv spec compliance: * Implement typed Observation, Action, and Reward Pydantic models.
   * step(action) → returns observation, reward, done, info.
   * reset() → returns initial observation.
   * state() → returns current state.
   * openenv.yaml with metadata.
   * Tested via openenv validate.
 * Minimum 3 tasks with agent graders: Each task defines a concrete objective with a programmatic grader (deterministic success/failure criteria).
Non-Functional Requirements
 * Hugging Face Space: Environment must run as a containerized HF Space tagged with openenv.
 * Containerization: Must include a working Dockerfile. The environment should start cleanly with docker build + docker run.
 * Documentation: README must include: environment description/motivation, action and observation space definitions, task descriptions (with difficulty), setup/usage instructions, and baseline scores.

This Is How Judging Works

Phase 1: Pass/Fail Gate
- HF Space deploys
- OpenEnv spec compliance
- Dockerfile builds
- Baseline reproduces
- 3+ tasks with graders

Phase 2: Agentic Evaluation
Scored
- Baseline agent re-run
- Standard Open LLM agent (e.g., Nemotron 3 Super) run against all environments
- Score variance check

Phase 3: Human Review
Top submissions reviewed by Meta and Hugging Face engineers for:
- Real-world utility
- Creativity
- Exploit checks

Disqualification Criteria
- Environment does not deploy or respond
- Plagiarized or trivially modified existing environments
- Graders that always return the same score
- No baseline inference script
